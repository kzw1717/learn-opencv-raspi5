## 1.2 なぜ「画像＝数字」から始めるのか

私たちは写真を見たとき、「葉っぱ」「鉢」「マット」という**意味**を感じ取ります。
でもコンピュータには、意味は最初まったく見えていません。コンピュータにとって画像は、

> 「明るさを表す数字が、縦と横にびっしり並んだ表」

でしかありません。この数字の表を計算で加工して、最終的に「マットが何パーセント覆われたか」という別の数字を取り出す——それが `leaf_monitor.py` がやっていることの正体です。

つまり OpenCV の関数はすべて、**数字の表を別の数字の表に変える道具**だと考えると、後の章の見通しが一気に良くなります。この感覚を、まず手を動かして体に入れていきましょう。

## 1.4 画像を読み込んで「形」を見る

では Python を書きます。`step1.py` というファイルを作りましょう。

```python
# step1.py
import cv2

# 画像をファイルから読み込む（中身は数字の表になって返ってくる）
img = cv2.imread("sample.jpg")

print(type(img))   # 何という種類のデータか
print(img.shape)   # 表の「形」＝大きさ
print(img.dtype)   # 1マスに入っている数字の種類
```

ターミナルで実行します。

```bash
python3 step1.py
```

出力はこんな感じになります（数字は撮った写真によって変わります）。

```
<class 'numpy.ndarray'>
(1520, 2028, 3)
uint8
```

1行ずつ読み解きます。

- `numpy.ndarray`：読み込んだ画像の正体は **NumPy の配列**、つまり数字の表です。普段 WSL2 で NumPy を触っているなら、まさにあれです。「画像が来た」のではなく「数字の表が来た」と考えてください。
- `(1520, 2028, 3)`：表の形です。**縦 1520 行 × 横 2028 列**、そして各マスに **3 個** の数字が入っている、という意味です。順番が「縦, 横」であって「横, 縦」でないことに注意してください。
- `uint8`：1つ1つの数字は **0〜255 の整数**です（`u`＝符号なし、`int8`＝8ビット整数）。0 が一番暗く、255 が一番明るい、という明るさの目盛りです。

3 という数字が出てきたのは、カラー画像が「赤・緑・青」の3つの明るさの組み合わせでできているからです。次の節でここを詳しく見ます。

> **やってみよう 1**
> `img.shape` の3つの数字のうち、画像を縦に拡大・縮小したら変わるのはどれでしょうか? 撮るときのカメラの解像度を変えると、どの数字が動くか予想してから確かめてみましょう。

## 1.5 1マスをのぞいてみる（画素＝ピクセル）

表の1マスを「画素（ピクセル）」と呼びます。たった1マスの中身を実際に見てみましょう。

```python
# step2.py
import cv2

img = cv2.imread("sample.jpg")

# 上から100行目、左から200列目の画素を取り出す
pixel = img[100, 200]
print(pixel)
```

実行すると、たとえばこう出ます（3つの整数が並びます）。

```
[52 138 90]
```

この `[52, 138, 90]` が、その1点の色を表す数字です。**3つの数字で1色**ということですね。スマホの写真も Web の画像も、突き詰めればこの「3つ組の数字」が何百万個も並んでいるだけ、というのがデジタル画像の正体です。

> **やってみよう 2**
> `img[0, 0]`（左上の角）と `img[-1, -1]`（右下の角）の画素を表示して、数字を比べてみましょう。明るい場所と暗い場所では、数字がどう違うでしょうか?

## 1.6 初心者が必ずハマる罠：OpenCV は BGR

さて、ここがこの章で一番大事な注意点です。

さきほどの `[52, 138, 90]` の3つの数字。普通に考えると「赤・緑・青（RGB）の順かな?」と思いますよね。**ところが OpenCV は順番が逆で、青・緑・赤（BGR）の順**で数字を持っています。

- `pixel[0]` → **B**lue（青）の明るさ
- `pixel[1]` → **G**reen（緑）の明るさ
- `pixel[2]` → **R**ed（赤）の明るさ

これを知らずに「0番目は赤」と思い込むと、色の判定がまるごと狂います。実際に、間違って RGB のつもりで色を入れ替えるとどうなるか、目で確かめてみましょう。

```python
# step3.py
import cv2

img = cv2.imread("sample.jpg")

# わざと青(B)と赤(R)を入れ替えてみる
swapped = img[:, :, ::-1]   # 3つ組の並びを逆さまにする

cv2.imwrite("swapped.jpg", swapped)
print("swapped.jpg を保存しました。VS Code で開いて比べてみてください。")
```

`sample.jpg` と `swapped.jpg` を VS Code で並べて開いてみてください。空が赤くなったり、肌が青くなったりして、**色が壊れている**はずです。これが「順番を間違えるとどうなるか」の正体です。

実はこの BGR の罠は、`leaf_monitor.py` の作者も強く意識していて、わざわざ「カメラの画像はいったん JPEG ファイルに保存してから読み込む」という回り道をしています。これは picamera2（カメラ用ライブラリ）が返す色の順番がややこしく、直接渡すと色が入れ替わる事故が起きやすいためです。**色の順番は、画像処理では命綱**だと覚えておいてください。

> **やってみよう 3**
> `swapped.jpg` をもう一度 `step3.py` と同じやり方で並べ替えたら、元の色に戻るでしょうか? なぜそうなるか、理由も考えてみましょう。

## 1.7 「数字の表」をいちばん実感できる方法：グレースケール

カラーだと「3つ組」で少し複雑です。色を捨てて**明るさだけ**にすると、画像が本当にただの数字の表だと実感できます。

```python
# step4.py
import cv2

img = cv2.imread("sample.jpg")

# カラー(BGR) → 白黒(明るさだけ)に変換
gray = cv2.cvtColor(img, cv2.COLOR_BGR2GRAY)

print(gray.shape)        # 形が変わる
print(gray[100, 200])    # 1マスの中身も変わる

cv2.imwrite("gray.jpg", gray)
```

実行すると、形がこう変わります。

```
(1520, 2028)
185
```

カラーのときは `(1520, 2028, 3)` で、各マスに3個の数字がありました。白黒にすると `(1520, 2028)` になり、**各マスの数字が1個**になります。色がなくなり、純粋に「明るさ（0〜255）の表」になったわけです。`gray.jpg` を VS Code で開けば、白黒写真として確認できます。

ここで初めて `cv2.cvtColor`（色空間の変換）という関数が登場しました。これは**第2章の主役**です。第2章では同じ `cv2.cvtColor` を使って、白黒ではなく「HSV」という、色を扱うのにとても便利な形へ変換していきます。`leaf_monitor.py` がマットの色を見分けられるのは、この変換のおかげです。

> **やってみよう 4**
> `gray.jpg` の明るいところ・暗いところで、`gray[行, 列]` の数字がどう違うか、いくつかの場所で確かめてみましょう。「明るい＝大きい数字」が体感できるはずです。

## 1.9 この章のまとめ

- 画像の正体は **NumPy の配列（数字の表）**。OpenCV の関数は「数字の表を別の表に変える道具」と考えるとよい。
- `cv2.imread` で読み込み、`.shape` で形（縦・横・色数）、`.dtype` で数字の種類（0〜255 の `uint8`）が分かる。
- 1マス＝画素は、カラーでは **3つ組の数字**。順番は **BGR（青・緑・赤）** で、ここを間違えると色が壊れる。
- 画面のない Pi では `cv2.imshow` は使えない。**`cv2.imwrite` で保存 → VS Code で開く**のがこのテキスト共通の確認方法。
- `cv2.cvtColor` で白黒にすると、画像が「明るさの表」だと実感できる。同じ関数が次章で HSV 変換に使われる。
